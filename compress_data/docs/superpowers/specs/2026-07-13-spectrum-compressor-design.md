# 光谱压缩串口上位机设计

## 项目目标

本项目交付一个基于 Python 和 PyQt6 的光谱压缩串口上位机。
程序支持 Windows 和 Linux，并适合在 VSCode 中开发、测试、运行和打包。
程序面向 7×24 小时连续运行，任何单个文件或单次串口发送失败均不得导致界面卡死或整体退出。

## 功能范围

程序提供完整的 PyQt6 图形界面。
用户可以选择输入目录和输出目录，并自动记忆上次配置。
用户可以选择串口、波特率、数据位、校验位和停止位。
界面提供刷新串口、打开或关闭串口、启动或停止任务、实时状态和运行日志。

程序监控输入目录中的 `.txt` 文件。
文件按首次发现顺序编号，并从每组 10 个文件中处理第一个文件，即第 1、11、21 个文件依次类推。
未被抽样选中的文件也记录到持久化状态中，确保程序重启后编号不发生漂移。

程序监控输出目录中的压缩文件。
待发送文件进入持久化发送队列，并按顺序通过串口发送。
相邻文件的发送开始时间至少间隔 1 秒。

## 架构

项目采用分层模块化设计。
`ui` 层负责 PyQt6 窗口、控件、信号连接和状态展示。
`application` 层负责启动和停止任务、协调后台服务以及向界面发布事件。
`compression` 层负责光谱解析、点位选择、强度舍入和十六进制编码。
`serial_comm` 层负责串口配置、连接管理、数据发送和失败重试。
`monitoring` 层负责目录轮询、文件稳定性判断和任务入队。
`persistence` 层使用 SQLite 保存文件顺序、压缩状态和发送状态。
`config` 层使用 JSON 保存用户界面和串口设置。

压缩模块不导入 PyQt6。
这表示算法可以独立测试和复用，不表示项目不使用 PyQt6。
PyQt6 后台服务会调用压缩模块，并通过信号把结果安全地发送给界面。

## 线程模型

Qt 主线程只执行界面操作。
目录扫描与压缩工作在独立后台线程中运行。
串口对象由唯一的串口后台线程创建和使用，避免跨线程访问冲突。
模块之间通过线程安全队列和 Qt 信号传递消息。

任务停止时，后台线程收到停止事件并在当前安全点退出。
程序关闭时，控制器依次停止监控、停止串口发送、关闭串口并等待线程结束。
所有后台循环在任务边界捕获异常并记录日志。

## 输入文件发现和抽样

监控服务默认每秒扫描一次输入目录。
纯轮询在 Windows 和 Linux 上行为一致，并避免文件系统事件丢失造成漏处理。
扫描结果仅接受普通 `.txt` 文件，并忽略临时文件和项目自身状态文件。

新文件以首次发现时间排序。
同一扫描周期内出现的文件使用可用的文件创建时间排序，并用文件名作为稳定的最终排序条件。
Linux 无可靠创建时间时使用修改时间作为排序依据。

每个新文件在 SQLite 中获得单调递增序号。
序号满足 `(序号 - 1) % 10 == 0` 时进入压缩流程。
其他文件标记为未抽样，之后不再重复计数或处理。

程序连续两次扫描文件大小和修改时间均未变化后，才认为文件写入完成。
读取过程中再次变化或读取失败时，文件保持等待状态并在后续扫描重试。

## 光谱解析和压缩

输入文件每条有效记录包含波数和强度两列。
解析器接受英文逗号、中文逗号和空白字符分隔。
解析器允许表头和空行，但会记录并跳过不能解析的数据行。
文件没有任何有效数据时，压缩任务失败并等待人工检查，不生成空输出文件。

算法在原始数据顺序上工作。
目标波数为 200、800、1400、2000、2600、3200、3800 和 4400。
只有位于文件实际波数范围内的目标波数参与定位。
每个目标波数选择绝对距离最小的数据点，并保留该点及其前后各 10 个点。
距离相同时选择原始顺序靠前的数据点。

非重点区域按原始索引每 30 个点保留 1 个。
所有重点区索引和降采样索引合并去重。
首点和尾点始终保留。
输出顺序与原始文件顺序一致。

保留点只输出强度值。
强度使用十进制 `ROUND_HALF_UP` 规则取整，即绝对值达到半数时向远离零的方向进位。
整数范围限定为 32 位有符号整数。
超出范围会使当前文件失败并写入明确日志。

每个整数编码为 32 位大端补码，并写成一行 8 位大写十六进制字符。
例如强度 300 输出为 `0000012C`，强度 -1 输出为 `FFFFFFFF`。
输出采用 UTF-8 文本并以换行符分隔，文件名与输入文件相同。
输出文件先写入同目录临时文件，再使用原子替换发布，防止发送器读取半成品。

## 串口发送

串口模块使用 pyserial。
界面支持串口、波特率、数据位、校验位和停止位配置。
默认参数为 115200 波特率、8 数据位、无校验和 1 停止位。

发送器逐行验证输出文件中的十六进制文本。
验证成功后使用 `bytes.fromhex()` 转为原始二进制字节，并按文件顺序发送。
程序不发送十六进制字符本身的 ASCII 字节。
协议不额外添加帧头、帧尾、长度字段或校验码，以免引入未经设备定义的协议。

相邻文件的发送开始时间至少间隔 1 秒。
一次写入后调用串口刷新，只有全部字节成功写入才标记为已发送。
失败任务保留在队首，并采用有上限的指数退避自动重试。
串口关闭或断开时发送队列暂停，串口重新打开后继续。

发送状态持久化到 SQLite。
程序异常退出后，未确认完成的文件会重新发送。
由于设备没有回执协议，程序只能以本地串口写入成功作为完成依据。

## UI 设计

主窗口使用清晰的单页布局。
顶部区域显示输入目录和输出目录，并提供浏览按钮。
中部串口区域提供端口刷新和所有串口参数下拉框。
操作区域提供串口开关和任务启停按钮。
状态区域显示任务状态、串口状态、已发现数量、已压缩数量、待发送数量、发送成功数量和最近错误。
底部区域显示带时间与级别的滚动日志，并提供清空日志按钮。

运行期间会禁用可能破坏当前任务一致性的目录设置。
串口打开期间会禁用串口参数编辑。
窗口关闭前执行有序停止，并在超时仍未结束时给出用户提示。

## 配置、状态和日志

用户配置保存在操作系统推荐的用户配置目录中。
配置内容包括输入目录、输出目录、串口参数、扫描间隔和窗口几何信息。
配置文件损坏时程序使用默认值启动，并将错误写入日志。

SQLite 数据库默认放在用户应用数据目录中。
数据库启用 WAL 模式、忙等待超时和短事务。
数据库表对规范化文件路径建立唯一约束，以避免重复发现。

日志同时写入界面和滚动日志文件。
界面只保留有限行数，避免连续运行造成内存增长。
日志包含启动停止、文件发现、文件稳定性、抽样决定、压缩结果、串口状态、发送重试和未捕获异常。

## 异常处理

单个输入文件格式错误只影响该文件。
单个输出文件十六进制错误只影响该文件。
串口断开不会停止目录监控和压缩。
目录暂时不可访问时后台服务记录限频日志并持续重试。
后台异常通过 Qt 信号报告给界面，不直接操作控件。

应用安装全局异常处理器记录主线程未捕获异常。
后台循环为每个任务设置异常边界。
程序避免无界队列、无界日志控件和无限快速重试。

## 测试策略

压缩模块使用 pytest 测试解析、目标波数邻域、每 30 点降采样、首尾保留、去重、舍入、负数和越界行为。
目录监控测试覆盖稳定性判断、首次抽样、第 11 次抽样、程序重启后的序号连续性和重复文件过滤。
持久化测试使用临时 SQLite 数据库。
串口测试使用 pyserial 的虚拟或注入式串口对象，覆盖字节转换、文件间隔、重试和顺序保持。
Qt 测试覆盖窗口创建、控件状态切换和后台信号更新。

最终验证包括完整测试套件、Python 编译检查、模块导入检查和无显示环境下的窗口创建检查。

## 验收标准

用户可以仅通过 UI 完成目录选择、串口配置、串口开关和任务启停。
程序准确处理第 1、11、21 个输入文件，并在重启后保持相同计数。
程序不会读取仍在写入的文件，也不会重复压缩已记录文件。
压缩结果符合目标波数邻域、每 30 点降采样、首尾保留和十六进制规则。
发送器按队列发送原始二进制字节，并保持至少 1 秒文件间隔。
单任务失败会记录、重试或隔离，但不会导致主界面卡死或程序整体退出。
项目可在 Windows 和 Linux 上安装、运行、测试和分别打包。
